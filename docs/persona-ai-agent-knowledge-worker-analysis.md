# ペルソナ起点分析（再定義版）

## 1. 目的

本ドキュメントは、`kra` の主対象を
「実装を含む多様な仕事を、AIエージェントで進める実務担当者」と定義し、
価値の中心を以下に置く。

1. 仕事の遂行速度
2. 大量アウトプットの追跡容易性
3. 後から説明できる証跡の蓄積

## 2. 想定ペルソナ（Primary Persona）

### 名前（仮）

**AI-Agent Driven Knowledge Worker**

### プロファイル

- 役割: 実装者/調査者/インシデント対応者/企画定義者を横断する実務担当
- 日常業務:
  - 実装
  - リポジトリ単体・横断調査
  - メトリクス分析（例: Datadog 等のMCP経由分析）
  - インシデント調査
  - 方針/コンセプトの定義
- 前提:
  - Jira などの外部チケット管理ツールを利用している
  - AIエージェントの出力が大量になりやすい
  - 仕事の痕跡を後で再利用・説明する必要がある

### 成功条件（このペルソナにとって）

1. チケットからすぐ仕事を開始できる
2. 中断後に短時間で文脈復元できる
3. 何をやったかを後で説明できる
4. 評価・職務経歴書作成に転用できる形で成果を取り出せる

## 3. 現状の `kra` が提供できている価値

1. workspace lifecycle（create/go/close/reopen/purge）が揃っている
2. repo 操作や診断（doctor）があり、ローカル運用の安全性が高い
3. spec-first とテストにより挙動の再現性が高い

一方で、次の価値が不足している。

1. AIエージェント実行の証跡化
2. 大量アウトプットの再検索・再利用
3. 日次/週次での説明可能な要約生成

## 4. 体験ギャップ（Job To Be Done 観点）

### JTBD-1: 「外部チケットから、すぐ着手したい」

- 現状課題:
  - チケット情報とローカル作業の橋渡しが弱い
- 必要機能:
  - チケット起点のコンテキスト初期化（作業対象/参照情報/実行履歴）

### JTBD-2: 「中断後に、迷わず再開したい」

- 現状課題:
  - `go` は移動に強いが、進捗文脈の復元が手作業
- 必要機能:
  - セッション要約と次アクションの自動記録

### JTBD-3: 「大量の成果を後で再利用したい」

- 現状課題:
  - 調査ログや判断理由が散逸し、再利用しづらい
- 必要機能:
  - 重要知見だけを保存する `worklog/insights` と横断再利用導線

### JTBD-4: 「評価/報告用に説明可能な形へ変換したい」

- 現状課題:
  - 実績はあるが、説明可能な形式で取り出せない
- 必要機能:
  - 日次/週次/期間指定サマリ生成

## 5. 非ゴール（明示）

1. `kra` を Jira の代替タスク管理ツールにすること
2. 複雑なタスク依存関係グラフを `kra` 内で完結管理すること
3. GUIホワイトボードの代替を目指すこと

## 6. 体験向上のために「あるべき機能」

以下は「ローカル仕事実行 + 追跡容易性」を中心にした優先度案。

## 6.1 P0（中核）

1. **Agent Skills Bootstrap（基盤）**
   - 内容:
     - `KRA_ROOT/.agent/skills` を SoT に固定
     - `.codex/skills` / `.claude/skills` を symlink 参照で統一
     - `kra bootstrap agent-skills` と `kra init --bootstrap agent-skills`
   - 効果:
     - エージェント運用の初期セットアップを標準化できる

2. **Flow-Oriented Skillpack（ツール提供）**
   - 内容:
     - 業務内容に踏み込みすぎないフロー中心スキルを公式提供
     - 調査・実行・要約の進め方を均質化
   - 効果:
     - ユーザーが手作業でスキルを作り込まなくても再現性を得られる

3. **Workspace Insight Capture（重要知見のみ）**
   - 内容:
     - 常時ログは取らない
     - 重要イベント時のみ会話内で保存提案し、承認時のみ保存
     - 保存形式は Markdown + frontmatter（`kind: insight`）
     - 保存先は `workspaces/<id>/worklog/insights/`
   - 効果:
     - 作業を阻害せずに高価値な証跡だけ残せる

4. **Resume Snapshot（再開特化）**
   - 内容:
     - 「前回ここまで」「未解決」「次の1手」を表示
   - 効果:
     - 中断復帰時間の短縮

5. **Evidence Pack（報告/評価向け）**
   - 内容:
     - 期間指定で成果を `Markdown/JSON` 出力
     - チケット更新文面や自己評価下書きを生成
   - 効果:
     - 報告コストを削減し、成果説明の質を上げる

## 6.2 P1（次段）

1. **Work Recall（検索/再利用）**
   - `worklog/insights` を横断検索し、再利用候補を提示

2. **Ticket Bridge（read-first）**
   - Jira等からチケット概要を取り込み、ローカル実行と紐付け

3. **Weekly/Monthly Narrative**
   - 週次/月次でのストーリー形式サマリ生成

4. **Session Timeline**
   - 時系列で作業判断と成果を可視化

5. **Cross-Workspace Insight**
   - workspace 横断の類似調査・再利用候補を提示

## 6.3 P2（後段）

1. **Lightweight Write-back**
   - チケットへコメント投稿など限定的な write-back

2. **Portfolio Export**
   - 職務経歴書素材への構造化エクスポート

## 7. UX 設計原則（このペルソナ向け）

1. **外部SoT尊重**
   - タスク詳細管理は Jira 等に委譲し、`kra` は実行/証跡に集中する

2. **入力省力化**
   - 作業中の常時記録は避け、必要時の承認保存を基本にする

3. **再開最適化**
   - 前回の文脈が1コマンドで復元できる導線を優先する

4. **説明可能性**
   - 「なぜその判断をしたか」が後から辿れること

5. **非対話前提**
   - CLI/AIエージェント/バッチ連携で同じ契約を使えること

## 8. 成果測定 KPI（導入効果を定量化）

1. チケット着手までの時間
2. 中断後の再開時間
3. 証跡欠落率（後から説明不能な作業の割合）
4. 週次報告作成時間
5. 過去作業の再利用率

## 9. 実装ロードマップ案

### フェーズ A（2-4 週間）

- Agent Skills Bootstrap
- Flow-Oriented Skillpack
- Workspace Insight Capture
- Resume Snapshot
- Daily Summary

### フェーズ B（4-8 週間）

- Work Recall
- Evidence Pack
- Weekly/Monthly Narrative

### フェーズ C（8 週間以降）

- Ticket write-back（限定）
- Portfolio Export
- Cross-workspace insight 高度化

## 10. まとめ

再定義後の主眼は、タスク関係の精密管理ではない。

`kra` は、外部チケット管理を前提にしつつ、
AIエージェントで進めた多様な仕事を
「速く進める」「後で辿れる」「説明できる」状態にするための基盤を目指す。

この方向性なら、GUIツールと競合せず、
CLIとしての強み（自動化、再現性、記録性）を活かして差別化できる。
